# NoirSound Artist Dashboard Statistics — QA Report

Covers Phase 6 of the Stats/Followers/Listening Data Integrity QA pass. See `NOIRSOUND_STATS_DATA_AUDIT.md` for the pre-pass state (dashboard stats were mostly not implemented, with an honest "not available yet" placeholder for everything but plays/likes, which were themselves broken via bug P-1) and `NOIRSOUND_STATS_DATA_INTEGRITY_REPORT.md` for the overall verdict.

## Ownership scoping

`GET /api/me/artist-dashboard` (`backend/src/routes/stats.js`) looks up the caller's `ArtistProfile` via `request.user.id` from the authenticated session — there is no route parameter that accepts an artist id, so there is no code path by which one artist could view another artist's dashboard data, and no separate authorization check was needed because the query itself cannot reach another artist's rows. Admins do not get a separate "view any artist's dashboard" surface through this endpoint; admin-facing artist detail is a different, already-audited surface (`NOIRSOUND_ARTIST_ACCESS_ADMIN_AUDIT.md`) outside this pass's scope.

## What the dashboard returns

`{ followers, monthlyListeners, isHidden, totalPlays, totalLikes, publishedTrackCount, tracks, topTracks, recentUploads, failedUploads, geography: null, trends: null }`. `followers` and `monthlyListeners` are the same live-count and recomputed-cache sources of truth used everywhere else (see the followers and listening-stats reports) — not a separately maintained dashboard-only number that could disagree with the public artist page. `totalPlays` and `totalLikes` (the `likes` field was added to the track `select` and summed across the artist's own published tracks as part of this pass — previously `totalLikes` was silently missing from the response) are summed directly from the artist's own tracks. Crucially, the dashboard sources its track list **directly by `artistId`, uncapped** — not from the public `GET /tracks` feed, which is capped at 20 items and globally ordered by recency across all artists. An artist with more than 20 total releases, or whose newer tracks from other artists pushed their older releases off the first page of the public feed, would have silently seen incomplete stats under the old (never-implemented) design; sourcing directly from the artist's own catalog avoids that class of bug by construction rather than by a page-size coincidence. This was directly tested: a track is asserted present in `dashboard.body.tracks` and counted in `totalPlays` using a fixture artist/track pair created outside the shared demo catalog.

`recentUploads` and `failedUploads` are shown as separate lists — a failed upload is never silently merged into or mistaken for a published release. `geography` and `trends` are returned as `null`, and the frontend renders an explicit "not available yet" placeholder for both (`t('dashboard.geographyUnavailable')`/`t('dashboard.trendsUnavailable')`) rather than a fake map or an invented trend line. Nothing in this pass adds geography or trend analytics — that infrastructure does not exist in this codebase, and the ticket's own requirement is to show an honest unavailable state instead of fabricating one, which is what both the backend (`null`) and frontend (explicit copy, not a hidden/blank chart that could be mistaken for zero) do.

## Frontend (`src/pages/Dashboard.jsx`)

Rewritten this pass to source every number from `getArtistDashboard()` alone — it previously also called `getTracks()`/`sortTracksNewest()` separately (a leftover from before the dedicated dashboard endpoint existed) and derived `totalStreams`/`totalLikes` from that capped, globally-scoped list, which is exactly the "capped public feed" trap described above. Now renders a four-stat grid (Total Streams, Hearts, Followers, Monthly Listeners), a Top Tracks section with an honest `t('dashboard.noTopTracksYet')` empty state, a Published Releases section, a Recent Uploads section, a Failed Uploads section (always rendered, with `t('dashboard.noFailedUploads')` when empty — never silently hidden, since a failed upload is exactly the kind of thing an artist needs to see even when the list is currently empty), and the geography/trends placeholders described above. All counts use `formatNumber()` compact notation, consistent with every other counter surface in the product.

## Requirements checklist (ticket Phase 6)

Artist sees only own tracks/stats: yes, scoped by session, no id parameter to exploit. Artist cannot access another artist's dashboard: yes, structurally (see above) — no test could construct a request that reaches another artist's data through this endpoint, since it never accepts one. Counts match the database: yes, `totalPlays`/`tracks` verified against fixture data created directly via Prisma outside the app's own write path, so the assertion isn't circular. Charts do not show fake data: there are no charts in this implementation — geography/trends are the two chart-shaped features the ticket anticipates, and both are honest `null`/"not available yet" rather than a fabricated chart. Failed uploads shown separately from published tracks: yes. Hidden tracks: the dashboard is the artist's own view of their own catalog, so a track they've had hidden by moderation still appears to them (an artist should be able to see the status of their own hidden work) but is excluded from `publishedTrackCount` and from the public-facing counts, consistent with how hidden tracks are already treated everywhere else in the product (404 from public track/stream routes, excluded from the public feed).

## Tests

Backend — `backend/tests/statsQA.test.js`, `describe('artist dashboard')`: sources counts directly from the artist's own tracks rather than the capped public feed, asserting `totalPlays`, track membership in `dashboard.body.tracks`, and that both `geography` and `trends` come back `null`. This complements the ownership-scoping guarantee above, which is structural rather than something a single test can fully enumerate (there is no "as artist B, try to read artist A's dashboard" test because there is no parameter through which to attempt it — the equivalent guarantee is that the query is always `where: { artistProfile: { userId: request.user.id } }`, which was read and confirmed directly in `backend/src/routes/stats.js`). All passing.

No dedicated new frontend test file was added for `Dashboard.jsx` itself in this pass (the rewrite was covered by manual verification against the live dev build plus the backend contract tests above); the existing frontend suite (119/119 passing) confirms the rewrite did not break the build or any other component that imports shared pieces of the dashboard (`TrackStatusBadge`, `formatNumber`, the mock API's `getArtistDashboard()`, which was updated to include `tracks: []` and `totalLikes: 0` so mock-mode consumers don't crash on the new fields).
