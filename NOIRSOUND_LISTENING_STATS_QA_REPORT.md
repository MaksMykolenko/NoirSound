# NoirSound Listening Data (Plays, Recently Played, User Stats) — QA Report

Covers Phases 3 and 5 of the Stats/Followers/Listening Data Integrity QA pass. See `NOIRSOUND_STATS_DATA_AUDIT.md` for bug P-1 (the original root cause) and `NOIRSOUND_STATS_DATA_INTEGRITY_REPORT.md` for the overall verdict.

## The qualified play rule

A play counts once the listener has actually listened for at least `min(30 seconds, 50% of the track's real duration)`. This is implemented once, server-side, in `qualifyThresholdSeconds()`/`isQualifiedPlay()` (`backend/src/lib/statsAccess.js`), and is always recomputed from the reported `durationListenedSeconds` versus the track's real `durationSeconds` — a client-sent `qualified` or `completed` flag is never trusted. This was directly verified with a test that sends `durationListenedSeconds: 1, completed: true, qualified: true` and confirms the response still comes back `qualified: false`. `PlayEvent.qualified` (`Boolean @default(false)`, added via migration `20260701203230_playevent_qualified_flag`) makes this a stored, queryable, first-class fact rather than something every consumer has to re-derive inconsistently — `Track.plays` incrementing, monthly-listener recomputation, recently-played, and listening-stats all filter on this one field.

`POST /api/tracks/:id/play-event` is rate-limited (60/minute per user-or-IP), creates a `PlayEvent` row on every call regardless of qualification (so the raw event log is complete for later analysis/repair), but only increments `Track.plays` and recomputes the artist's monthly listeners when `qualified === true`. Reported duration is capped at 3600 seconds server-side as a basic sanity bound against a malformed or malicious client.

## Frontend play tracking (`src/store/playerStore.js`)

Previously (bug P-1) the store reported a play immediately on every `audio.play()` call with a hardcoded `durationListenedSeconds: 0`, meaning the backend's own threshold could never be satisfied by real traffic, while still creating a full `PlayEvent` row on every click — a page load, a skip, a `next()`/`previous()` call, or a repeat-one loop all silently polluted recently-played and listening stats with zero-duration phantom events, and `Track.plays` could never increment through normal use at all.

The rewrite tracks real listened position via `audio.currentTime` on every native `timeupdate` event, takes the running maximum (so seeking backward never decreases the tracked position, and seeking forward to "game" the threshold doesn't create extra credit beyond the real position reached), and reports exactly one qualifying play per playback session — at the moment the threshold is first crossed, gated by a per-session `qualifyReported` flag that lives in module scope (so it survives React re-renders/remounts, which a component-local flag would not). `playTrack()` no longer sends anything on its own; it only resets the tracking state for the new track. This satisfies every requirement in the ticket's "a play should NOT be counted just because…" list: no count on page load, no count from social/crawler metadata fetches (those never touch the player at all), no double-count from React remounts (module-scope state), no double-count from progress polling (single-report gate), and starting the same track again after a qualifying play starts a fresh session that can qualify again (verified by a test that replays the same track and expects exactly two total reports, not zero and not more than two).

Bot/crawler and `/stream` HEAD-request traffic never reaches this code path at all — it lives entirely in the browser player, not the stream endpoint, so a crawler fetching Open Graph metadata or issuing a HEAD request to preview a link cannot generate a play event by construction, independent of any user-agent sniffing.

## Recently played and user listening stats

`GET /api/me/recently-played` and `GET /api/me/listening-stats` both filter to `qualified: true` only. Recently-played deduplicates by track (latest valid play per track, most recent first). Listening stats (`GET /api/me/listening-stats`) computes `topTracks` (grouped by track, with the full track object nested per entry) and `topArtists` (grouped by artist, flat summary shape — these two are intentionally different response shapes, documented so a future consumer doesn't assume symmetry), plus `topGenres` (using the shared `normalizeGenre()` taxonomy), `totalListeningSeconds`/`totalListeningMinutes`, `tracksPlayed`, `uniqueArtists`, and `topGenre`. `topTracks`/`topArtists` were previously dead plumbing — the frontend already read `stats.topArtists`/`stats.topTracks` but the backend never populated them — and are implemented (not just documented as missing) in this pass.

Privacy: these two endpoints are scoped to `request.user.id` from the authenticated session — there is no route or parameter that accepts another user's id, so there is no code path by which one user's listening history can be read by another. This was directly tested (see below).

`src/components/profile/ListeningStats.jsx` renders `topTracks`/`topArtists` as clickable rows (navigating to the track/artist page), using `formatNumber()` for play counts and honest `t('stats.notEnoughData')` empty states when either array is empty — no placeholder rows, no fabricated entries. `ListeningStats.jsx` already correctly labeled "playback starts" versus "measured listening time" and hid percentage breakdowns when the sample size is too small to be meaningful (pre-existing, unchanged this pass); the new sections follow the same honesty standard.

## Tests

Backend — `backend/tests/statsQA.test.js`, `describe('qualified plays')`: a play below threshold does not increment `Track.plays` or `monthlyListeners`; a play that crosses the threshold increments both; a forged `qualified: true`/`completed: true` with a 1-second reported duration is still recomputed as unqualified; an artist cannot inflate their own monthly-listener count by replaying their own track (`Track.plays` still increments — it is a real play — but `monthlyListeners` stays 0, since it excludes the artist's own plays of their own tracks); an unqualified attempt never appears in recently-played or listening-stats, while a subsequent qualifying attempt on the same track does, with the correct `playCount`. All passing.

Frontend — `src/store/__tests__/playerStore.test.js` (rewritten, 11 tests): starting playback alone reports nothing and touches neither the API nor Recently Played; crossing the threshold reports exactly one qualifying play and adds to Recently Played exactly once; skipping before the threshold reports nothing; replaying the same track starts a fresh session that can qualify again (2 total reports across 2 sessions); plus a dedicated `describe('qualifyThresholdSeconds')` block testing the pure function directly (caps at 30s for a 180s track, uses 50% = 20s for a 40s track, falls back to 30s for a zero/undefined duration); plus the pre-existing stream-error, like, and queue-management tests, unchanged. `tests/components/ListeningStats.test.jsx` (4 tests, extended this pass): renders real `topTracks`/`topArtists` from the backend rather than a placeholder, and shows the honest empty state (not fabricated entries) when both arrays are empty, on top of the pre-existing "no fabricated metrics" and "no invented listening time" tests. All passing.
